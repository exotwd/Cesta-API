# App integration: foot crossings and walking transfers

Foot crossings are returned by journey search as ordinary journey legs with no public-transport
trip attached. They can mean a nearby-stop walk, an in-station platform crossing, or a same-station
interchange generated by the routing backend.

The mobile app should not calculate these transfers itself. It should render the legs returned by
the API in order.

## Request

Use the existing journey search endpoint:

```http
POST /journeys/search
Content-Type: application/json
```

```json
{
  "from": {"type": "city", "id": "city:CZ:554782"},
  "to": {"type": "city", "id": "city:CZ:544256"},
  "datetime": "2026-07-19T10:30:00+02:00",
  "mode": "depart_at",
  "transport_modes": ["metro", "tram", "bus", "train"],
  "max_transfers": 4,
  "walking_speed": "normal",
  "prefer_reliable_transfers": true,
  "offline_compatible": false
}
```

`walking_speed` affects backend walking durations for nearby-stop access links. Supported values are
`slow`, `relaxed`, `accessible`, `normal`, and `fast`; unknown values are treated as normal. Store
the user's mobility preference and send it on every search instead of scaling walking times in the
app.

When opening journey detail, repeat the same search with `include_intermediate_stops: true` if the
screen also needs train/bus intermediate calls. Walking legs do not have `stop_calls`.

## Detecting a foot crossing

A leg is a foot crossing when both identifiers are absent:

```dart
bool isFootCrossing(JourneyLeg leg) {
  return leg.routeId == null && leg.tripId == null;
}
```

Example API leg:

```json
{
  "from_stop_id": "ggu_czptt_gtfs_latest:-SR70S-CZ-33722-2",
  "to_stop_id": "ggu_czptt_gtfs_latest:-SR70S-CZ-33722-7",
  "route_id": null,
  "trip_id": null,
  "departure_time": 65220,
  "arrival_time": 65520,
  "mode": "unknown",
  "warnings": ["walking_transfer:0"]
}
```

Use `arrival_time - departure_time` as the crossing duration. The `walking_transfer:N` warning is
the distance in meters when known. `0` means the backend did not have a reliable distance, often for
same-station/platform interchanges; do not hide the leg just because the distance is zero.

## Flutter model

Extend the existing journey models:

```dart
class Journey {
  final List<JourneyLeg> legs;
  final int walkingDistanceMeters;

  Journey.fromJson(Map<String, dynamic> json)
      : legs = (json['legs'] as List)
            .map((item) => JourneyLeg.fromJson(item))
            .toList(),
        walkingDistanceMeters = json['walking_distance_meters'] ?? 0;
}

class JourneyLeg {
  final String fromStopId;
  final String toStopId;
  final String? routeId;
  final String? tripId;
  final int departureTime;
  final int arrivalTime;
  final String mode;
  final List<String> warnings;

  JourneyLeg.fromJson(Map<String, dynamic> json)
      : fromStopId = json['from_stop_id'],
        toStopId = json['to_stop_id'],
        routeId = json['route_id'],
        tripId = json['trip_id'],
        departureTime = json['departure_time'],
        arrivalTime = json['arrival_time'],
        mode = json['mode'],
        warnings =
            (json['warnings'] as List? ?? const []).map((item) => '$item').toList();

  bool get isFootCrossing => routeId == null && tripId == null;

  int get durationSeconds => arrivalTime - departureTime;

  int? get walkingDistanceMeters {
    final warning = warnings.firstWhere(
      (item) => item.startsWith('walking_transfer:'),
      orElse: () => '',
    );
    if (warning.isEmpty) return null;
    return int.tryParse(warning.split(':').last);
  }
}
```

## Display rules

1. Render foot crossings inline between transit legs, not as separate journey cards.
2. Use a walk/change icon and a dashed connector in timeline views.
3. If `walkingDistanceMeters` is greater than zero, show distance and duration, for example
   `Walk 180 m, 3 min`.
4. If distance is missing or zero and both stop names resolve to the same station, show
   `Change platforms, 5 min`.
5. If distance is missing or zero but the stop names differ, show `Walk/change, 5 min`.
6. Keep the returned order of legs. Do not merge adjacent transit legs across a crossing.
7. Do not discard a journey because it contains a zero-distance crossing.
8. Use top-level `journey.walking_distance_meters` only for summary badges. Use per-leg duration
   for timeline timing.
9. Keep times as service-day seconds. Values may exceed 86400 after midnight; format them with the
   journey service date instead of modulo-only labels.
10. Always show top-level API `warnings`; they explain legacy calendar fallback, skipped next-day
    warmups, or removed invalid candidates.

## Map rendering

Use the backend's stop data as the source of truth:

- If the response includes related stop coordinates or `stop_calls`, draw a dashed straight segment
  between the crossing endpoints.
- If coordinates are missing, omit the map segment and keep the timeline row.
- Do not snap to roads or pedestrian paths in the app unless a future API response includes an
  explicit walking geometry.
- For same-station/platform crossings, prefer a compact station interchange row over a visible map
  line.

## Accessibility

Add a user setting for walking pace or reduced-mobility routing:

- `accessible` or `slow`: use when the user needs more time for crossings.
- `normal`: default.
- `fast`: only when explicitly selected.

The backend will return different transfer feasibility and durations based on this value. The app
should not locally shorten the returned transfer time.

## QA checklist

- A journey with `route_id: null` and `trip_id: null` renders as a walk/change row.
- Zero-distance same-station crossings are visible and do not disappear.
- A faster transfer journey is not hidden just because a direct journey also exists.
- Journey detail preserves separate legs before and after the crossing.
- The selected journey is matched by ordered leg `trip_id`, departure time, and arrival time, not by
  `journey-N`.
- Offline/cache layers preserve `warnings`, null `route_id`, null `trip_id`, and
  `walking_distance_meters`.
